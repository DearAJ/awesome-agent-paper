# MEME — 演化记忆基准：级联失效（Cascade）是全领域的谷底

> **arXiv**：2605.12477（2026.05）｜**机制**：**benchmark**（演化记忆 / 依赖失效）
> **HF 月榜**：2026-05，7↑｜**机构**：KAIST AI
> **关键词**：evolving memory · multi-entity · Deletion · Cascade · Absence · dependency reasoning
> **一句话定位**：测"更新应**传播/失效依赖项**"的演化记忆基准——**所有系统在 Cascade ~3%、Absence ~1% 崩溃**，把"级联失效"这个全领域硬骨头量化到谷底。

---

## 1. 这篇论文为什么重要

**一句话**：记忆更新最难的不是"改对当前这条"，而是"**改了 X 之后，依赖 X 的下游记忆/结论要不要跟着失效**"；MEME 把这个**级联失效（cascade / dependent invalidation）**做成可测基准，发现**所有系统都在这上面崩**（Cascade ~3%、Absence ~1%）。

它是本专题**评测侧最关键的一篇**，也是 ReTrace 的**靶心评测**：

- **把"级联失效"从口头难点变成可测数字**：~3% 是个**触目惊心的低分**，说明"更新传播到依赖项"是当前记忆系统的**普遍盲区**。
- **它和参数侧的 RippleEdits/MQuAKE 说的是同一件事**：知识编辑栽在"编辑不传播多跳"，agent 记忆栽在"删除不传播依赖"——**记忆侧与参数侧在同一个洞里**（见 `00-summary` 趋势 2）。
- **ReTrace 若能在 MEME 的 Cascade 上显著提分，就是杀手锏结果**：因为你"回溯到矛盾源、**失效依赖该 evidence 的下游 summary**"正是为这个洞设计的。

它与 [[02-stale]] 的 propagation-aware search、[[01-memforest]] 的"受影响路径"局部更新都指向同一问题——**更新如何正确传播**。

## 2. 核心方法

### 2.1 诊断：静态检索 OK，依赖失效崩

| 能力 | 现状 |
| --- | --- |
| 静态检索（retrieval） | 大多数系统能做 |
| **更新 → 失效依赖项** | 🔴 几乎全崩 |

> 核心：现有记忆评测多测"能不能取回",但**真实更新要求"改了 X→依赖 X 的 Y 自动失效/更新"**——这个维度此前没被系统量化。

### 2.2 MEME 的三类演化任务

| 任务 | 测什么 |
| --- | --- |
| **Deletion（删除）** | 一条记忆被删除后，系统是否反映**删除后的状态**（而非还用旧值） |
| **Cascade（级联）** | 更新 X 后，**依赖 X 的下游**是否跟着失效/更新（依赖推理） |
| **Absence（缺失）** | 当某信息**应当不存在**时，系统是否正确反映其缺失 |

外加 **multi-entity（多实体）**：在多个实体交织演化下测记忆，而非单实体孤立。

### 2.3 核心结果：依赖推理崩盘

$$
\text{Cascade} \approx 3\% \quad\quad \text{Absence} \approx 1\%
$$

—— 所有被测系统在**依赖失效**上接近**全错**。静态召回正常、一旦要求"更新传播到依赖项"立刻崩。

> 摘要未披露：被测系统清单、各系统逐项分数、任务构造细节——**需读 PDF**。

## 3. 关键实验结果

| 任务 | 最佳系统表现 | 含义 |
| --- | --- | --- |
| **Cascade（依赖失效）** | **~3%** | 级联失效几乎完全未解 |
| **Absence（应缺失）** | **~1%** | "正确反映缺失"同样崩 |
| Deletion / 静态召回 | 相对正常 | 问题专属于**依赖传播**，不是检索本身 |

> **核心信号**：**~3%** 不是"还有提升空间"，而是"**这个能力基本不存在**"。MEME 因此成为衡量"记忆更新是否真正传播"的**最严苛、最干净的基准**——也是 ReTrace 最该打的靶。

## 4. 对领域的影响 / 后续方向

### 🌟 学界影响

1. **把级联失效量化到谷底**：~3% 让"更新传播到依赖项"从模糊难点变成**可对标的硬指标**。
2. **统一记忆侧与参数侧的失败叙事**：MEME（记忆）与 RippleEdits/MQuAKE（参数）证明**同一个洞**——这是本专题"两侧合流"趋势的关键证据。
3. **multi-entity + evolving 的评测范式**：比单实体静态记忆更贴近真实长程部署。

### ⚠ 局限

- **是基准、非解法**：指出问题、量化谷底，但不给解决级联失效的方法（留给后续，如 ReTrace）。
- 任务构造的覆盖范围（哪些依赖类型）需读 PDF 确认。

### 🔮 趋势

1. **"更新传播"成为记忆评测的核心维度**（超越静态召回）。
2. **依赖结构化**：要解级联失效，记忆很可能需要**显式依赖图/链**（TMS 思路）——这正是 ReTrace 的方向。

### 📊 同方向工作

- [[02-stale]]：propagation-aware search 想解决的正是 Cascade。
- [[01-memforest]]：MemForest 有树/局部更新但**没测 Cascade**——MEME 正好暴露这类系统的盲区。
- **RippleEdits / MQuAKE**（参数侧）：级联失效的参数版，与 MEME 互为镜像。
- [[07-belief-revision]]：经典 TMS 的 dependency-directed backtracking 就是为级联失效设计的——MEME 给了它现代基准。

## 5. 与 ReTrace 的关系（重点）

| ReTrace 的设计点 | MEME 的对应 |
| --- | --- |
| 回溯到矛盾源节点 + **失效依赖该 evidence 的下游 summary** | 🔴 **正是 Cascade 测的能力** |
| change-log 记录 before/after | 支持 Deletion 后状态正确反映 |
| 加性 vs 非加性判定 | 决定何时触发级联失效 |

**结论**：MEME 是 ReTrace 的**主靶评测**。三点用法——

1. **主结果就打 Cascade**：当前 ~3% 是地板，ReTrace 的"回溯+依赖失效"若能把 Cascade 显著提上去，就是**最有冲击力的结果表**（远比再做一个 LoCoMo 召回数字有说服力）。
2. **它证明 ReTrace 解决的是真问题**：~3% 说明级联失效是全领域盲区——你不是在解决臆想的问题。
3. **它也是 ReTrace 的最大技术风险**：早先我提醒你"删 evidence 后下游 summary 依赖它"——MEME 的 ~3% 正是这个风险的全行业写照。ReTrace 必须**显式维护依赖关系**（哪条 summary 依赖哪些 evidence），矛盾回溯时**沿依赖链失效**——否则你也会落在 ~3%。这把 ReTrace 直接接到经典 **TMS（依赖网络 + dependency-directed backtracking）**（见 [[07-belief-revision]]）。
